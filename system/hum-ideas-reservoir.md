# Hum Ideas Reservoir

Proactive design layer on top of the reactive hum harvest. When the 14 live
harvesters surface nothing new, hum doesn't go silent — it pulls from a
reservoir of pre-computed ideas generated by a deep-thinking agent.

## Why

Hum's default design is reactive: scan situation → scan topics for deltas →
Sonnet ranks → send best. If harvest returns only stale repeats or
morning-digest-suppressed candidates, every tick ends silent. In practice
this produces ~1 pulse/day against a soft cap of ~4.

The fix is not lowering threshold (quality bar is right). It's giving Sonnet
more to work with — ideas the live harvesters can't produce because they
require synthesis across calendar + tasks + memory + taste, or lookahead into
the next 7-14 days.

## The goal: hearts and 👍 (reinforcement loop)

Every idea that becomes a pulse is a bet. The success signal is the user's
reaction — `heart` / `👍` / `helpful` / `more` are positive; `skip` /
`snooze` / `👎` are negative; silence is weakly-ambiguous. **The ideation
agent optimizes for ideas the user would react positively to** — not for
information density, novelty, or coverage breadth.

**Anti-sycophancy guard:** "would the user heart this?" is the success
criterion, but utility is the substance. An idea earns a heart because it
genuinely helps (new angle, unblock, timely prep, taste-aligned discovery),
not because it flatters. If an idea lane starts skewing toward "pleasant but
useless," the scorecard catches it: high hearts + low `more` + low follow-on
action is a red flag, not a win.

## Scorecard — `data/hum-ideas-scorecard.json`

Living aggregate of per-category and per-pattern reaction performance. Each
ideation run reads this first, then writes an updated version.

```json
{
  "updated_at": "...",
  "by_category": {
    "taste-match":      {"sent": 12, "heart": 5, "up": 3, "more": 2, "skip": 1, "down": 0, "silent": 1, "score": 0.78},
    "event-prep":       {"sent": 8,  "heart": 1, "up": 2, "more": 0, "skip": 2, "down": 1, "silent": 2, "score": 0.32},
    "stale-p1":         {"sent": 5,  "heart": 0, "up": 1, "more": 0, "skip": 3, "down": 1, "silent": 0, "score": 0.08},
    ...
  },
  "by_lens": {
    "pop-art-photography-closing": {...},
    "project-X-blocker-cleared":   {...},
    ...
  },
  "rejected_patterns": [
    {"pattern": "daily-task-summary", "skips": 4, "last": "..."},
    {"pattern": "p1-email-reminder",  "skips": 6, "last": "..."}
  ]
}
```

`score = (heart*1.0 + up*0.6 + more*1.2 - skip*0.8 - down*1.0) / sent`, with
Laplace smoothing (add 1 to denom) so small samples don't swing to extremes.
`more` is the strongest positive (explicit ask for deeper dive). Silence is
zero weight — could be a late reaction.

## Reinforcement at ideation time

Every `hum-ideation` run:

1. Reads `hum-ideas.jsonl` (current + historical consumed).
2. Reads `hum-feedback.jsonl` and joins reactions to consumed ideas via
   `ping_id`.
3. Rebuilds `hum-ideas-scorecard.json` with fresh aggregates.
4. Uses the scorecard to bias generation:
   - Target 50% of new ideas in top-2 categories by score (exploit)
   - Target 30% in untested or low-sample categories (explore)
   - Target 20% in bottom-scoring categories *only if* a specific
     reason-to-retry exists (e.g. "stale-p1 always fails as a reminder —
     try escalation-path-only variant this time")
   - Avoid any idea whose subject-signature near-matches
     `rejected_patterns`
5. Writes new ideas.

## Reinforcement at consumption time

When hum picks an idea and sends the ping:

1. Append a `consumed` record to `hum-ideas.jsonl` with the `ping_id`.

When `merlin hum feedback <ping-id> <reaction>` is called (chat-agent
handles this per existing rules):

2. The CLI checks if the ping originated from an idea (by looking up
   `ping_id` in `hum-ideas.jsonl` consumed records).
3. If yes, append a `reaction` record to `hum-ideas.jsonl` tying the
   reaction to the idea.
4. This closes the loop — next ideation run sees it.

## Silence handling

A pulse with no reaction within 6h is recorded as `silence` (existing hum
behavior in `hum-feedback.jsonl`). For ideas, silence contributes zero to
the score but is tracked separately — a category that's *always* silent is
a lurking problem (uninteresting-but-not-offensive). If `silence` rate on a
category exceeds 70% of sent with ≥5 samples, flag it in the scorecard for
manual review (never auto-demote).

## File — `data/hum-ideas.jsonl`

Append-only JSONL. Each idea:

```json
{
  "id": "idea-<yyyymmdd-hhmm-hash>",
  "created_at": "2026-04-15T18:00:00Z",
  "source": "hum-ideation-agent" | "manual",
  "category": "event-prep" | "stale-p1" | "weekend-planning" | "travel-prep"
           | "family" | "project" | "taste-match"
           | "cross-topic" | "stuck-task" | "seasonal-closing",
  "subject": "short phone-ping-ready subject (<60 chars)",
  "rationale": "why this is worth a pulse",
  "why_now": "time-sensitive context",
  "urgency": 0.4,
  "time_sensitive_until": "2026-04-22T00:00:00Z" | null,
  "draws_on": ["calendar", "tasks", "email-mem", "trip-history"],
  "supporting_data": { ... freeform ... },
  "consumed": false,
  "consumed_at": null,
  "ping_id": null
}
```

On consumption: **do not mutate the original line**. Append a second record
with the same `id` and `consumed: true, consumed_at, ping_id`. Readers take
the latest record per id.

## Harvester — new `ideas` topic

Added to `data/hum-interests.json` with low cadence (cheap to read).
Implementation in `hum-harvest-all.mjs`:

1. Read reservoir, collapse per-id to latest record
2. Filter out consumed or time-expired
3. Filter against recent `hum-feedback.jsonl` — skip ideas whose `category`
   or subject pattern was `skip`/`snooze`'d in last 7 days
4. Emit top 3 as candidates with their stored `urgency` × freshness boost

The `dedup_key` for idea candidates is the idea `id` — Sonnet sees it in
context. If Sonnet picks an idea candidate, the runner marks consumed and
records `ping_id`.

## Job — `hum-ideation`

New scheduled job. Dispatches the ops-agent (opus) to generate ideas.

**Schedule:** 06:00 and 14:00 local daily. Also: on-demand from
`hum-harvest-all.mjs` when reservoir has < 5 unconsumed ideas that clear
a minimum-urgency floor (0.35).

**Playbook** (`agent/ops-agent/jobs/hum-ideation.md`):

1. Read current reservoir. Read `hum-runs.jsonl` last 48h (what was
   pinged + what was rejected and why). Read `hum-feedback.jsonl` last
   30 days (what the user reacted to).
2. Pull broad context: calendar next 14 days, Google Tasks all open,
   active projects (from `user_*` / `project_*` wiki pages), the user's
   tastes, trip-history, recent photos.
3. For each of these lenses, propose candidate ideas:
   - Event prep: any calendar item in 1-5 days needing context, prep,
     logistics, or a decision
   - Stale P1 escalation: P1 emails pending >5 days — suggest escalation
     path, not just the reminder
   - Weekend planning: weekend openings + shared-time blocks +
     venues filtered by user preferences + closing-soon events
   - Travel prep: trip in <14 days → pre-flight prep, dining research,
     event alignment
   - Family / recurring people: milestones, gift prep starting 2mo out,
     birthday brainstorms — sourced from the user's contacts + tagged
     wiki pages
   - Taste-match: new openings / limited-run / matching the user's
     profile pages
   - Cross-topic synthesis: two weak signals that together make a strong
     idea (e.g., partner out of town + event opening + dog-friendly walk
     = a candidate solo-day plan)
   - Stuck tasks: tasks open >14d — propose an unblock
   - Seasonal closing: exhibits/events ending in <7d that match taste
4. Filter out anything dup of current reservoir or pinged in last 48h.
5. Write new records to `hum-ideas.jsonl`. Target 8-15 new ideas per
   run. Silent on zero-new-ideas (not an error).
6. Notify nothing. Hum will pick them up next tick.

**Boundaries** (same as hum):
- Read-only on all external systems
- Never sends email, iMessage, or Merlin pings directly
- Never modifies tasks, calendar, email
- Only write is appending to `hum-ideas.jsonl`

## Consumption tracking

On ping send (runner step 4 in `jobs/hum.md`), if the winning candidate came
from the `ideas` harvester:
```
echo '{"id":"<idea-id>","consumed":true,"consumed_at":"<now>","ping_id":"<ping-id>"}' \
  >> data/hum-ideas.jsonl
```

## Pruning

Weekly (Sundays via `hum-review` job) — prune ideas where:
- `consumed=true` AND `consumed_at` older than 14 days
- `time_sensitive_until` in the past by >7 days
- `created_at` older than 60 days and still unconsumed (never clicked → not
  interesting)

Write pruned file by rebuilding (not in-place edit). Keep ~last 200 records.

## Integration with existing hum rules

- Idea ping counts as one ping per run; daily cap unchanged
- Configured quiet windows (see `hum.md` § QUIET WINDOWS) still suppress
- Ideas don't bypass threshold — they compete at the Sonnet rank stage
- `urgency=high` ideas can pass soft quiet windows the same as live
  high-urgency candidates
