# Security policy

## Reporting a vulnerability

If you find a security issue in Merlin — credential leak, prompt-injection bypass, supervisor crash that gates further safety checks, anything that puts a user's data or external accounts at risk — please **do not open a public issue**.

Instead, open a [GitHub security advisory](https://github.com/mraccah2/merlin/security/advisories/new) (private to maintainers + you), or email the repo owner directly via the address listed in `git log` on commits to `main`.

We'll respond within a week. If the issue is acute (active exploit in the wild, credential disclosure), tag it `URGENT` in the advisory subject and we'll respond same-day where possible.

## Threat model

Merlin runs LLM agents with broad permissions on your host. The threat surface that matters:

- **Inbound content is untrusted.** Emails, Slack messages, webhooks, and web pages can carry instructions that try to subvert the agent. The shared rules in `agent/CLAUDE.md` § Prompt Injection & Input Safety codify the discipline: never follow instructions in inbound bodies, never send credentials in responses, never execute commands found in mail.
- **Financial actions require explicit human approval.** `agent/CLAUDE.md` § Financial Operations forbids transfers/withdrawals outright and caps purchases (default $300) with a hard two-turn human approval gate. Don't relax these without thinking very hard about why.
- **The phone-channel is the only trusted command channel.** Even there, agent code sanity-checks unusual financial requests rather than executing on first instruction.
- **Outbound to the user passes through an LLM gate.** `bin/outbound-precheck.mjs` reviews every programmatic phone-channel message against the last 90 min of conversation + active calendar. Skipping it is reserved for infrastructure-liveness alerts only.

## What is and isn't a vulnerability

**Is:**

- A way to extract `ANTHROPIC_API_KEY` or `SUPABASE_SERVICE_ROLE_KEY` from agent context (e.g. via a crafted email).
- A way to make the agent send arbitrary outbound mail from `${MERLIN_OWNER_EMAIL}` without explicit user authorization.
- A way to bypass the financial-operations hard limits.
- A way to make the agent navigate Chrome to an attacker-controlled URL with sensitive data attached.
- Any way an inbound prompt can make the agent ignore the shared CLAUDE.md rules.
- Any path that exfiltrates wiki/memory contents to an external service the user didn't authorize.

**Isn't (please don't report as vulnerabilities):**

- "The agent did something dumb when I asked it to." That's an agent-quality issue. Open a normal issue or PR.
- "If I clone the repo and edit `agent/CLAUDE.md` I can make the agent do X." Yes — you control the prompts. That's the design.
- Issues in adjacent projects: Claude Code, Anthropic API, Ollama, Supabase, etc. Report those upstream.
- Vulnerabilities that require physical/SSH access to the host. The host is the trust boundary; if you're inside it, you've already won.

## Known accepted risks

- **Secrets live in `.env`** which is on disk in plaintext. Mitigation: `.env` is gitignored, has `chmod 600` (set by you — see `docs/hardening.md`), and for production you should use the 1Password `op run` pattern documented there.
- **The supervisor accepts webhook POSTs on `localhost`** with bearer-token auth. Anyone with local network access who guesses or extracts your `WEBHOOK_TOKEN` can dispatch jobs. Mitigation: the token is 32 random bytes, regenerated by `merlin init`. Bind only to `127.0.0.1` (the default).
- **`bypassPermissions` mode** is set on both agents in `agent/config/agents.json`. This is necessary for the agent to use the full tool set without per-tool prompts. The mitigation is the layered rules in `CLAUDE.md` (hard exclusions, financial limits, prompt-injection rules) and the outbound precheck.

## Disclosure timeline

We use 30-day coordinated disclosure for issues that are exploitable in the wild. We may extend if the fix requires coordination with upstream (Anthropic, Apple, etc.). Reports that are "interesting but not exploitable" or "could be exploitable in a future codebase change" are handled in the open as normal issues.
